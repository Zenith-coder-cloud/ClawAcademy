## УРОК 1 — Твоя система стала продуктом
**track:** all | **timer:** 15 мин | **prev:** null | **next:** 2

### whyNeeded
В Blocks 1–3 ты строил систему для себя. Она работает, экономит тебе время, помогает с задачами. Это уже ценность. Но пока ты единственный кто ею пользуется — ты упускаешь большую часть потенциала. В этом блоке всё меняется: ты перестаёшь быть пользователем и становишься поставщиком. Это не усложнение — это следующий уровень той же игры.

### case
**Артём живёт в Новосибирске.** В Block 3 он собрал мультиагента который каждую ночь мониторит крипто-новости и присылает ему дайджест. Сосед по офису увидел и говорит: «я бы платил за такое». Артём думает: «но это же просто мой агент, это несерьёзно». Через месяц такой же мультиагент на Upwork продаётся за $800 разово плюс $200/мес за поддержку. Артём всё ещё думает.

### capabilities
- Понять три пути монетизации агента и выбрать свой
- Увидеть разницу между «агент для себя» и «агент для клиента»
- Получить первый список потенциальных клиентов или покупателей

### infoBlocks
**🔄 Три пути монетизации:**

**Путь 1: Виртуальный сотрудник** — ты настраиваешь агента под конкретного клиента, агент работает как его «сотрудник». Клиент платит тебе ежемесячно как за подрядчика. Твой контроль — максимальный, твой доход — стабильный ретейнер.

**Путь 2: ClawHub Marketplace** — ты публикуешь агента на ClawHub, любой пользователь OpenClaw может его купить или подписаться. Один раз создал — продаётся сам. Пассивный доход без клиентов.

**Путь 3: SaaS продукт** — ты строишь платформу где один агент обслуживает много клиентов одновременно, они платят подписку. Самый сложный путь, но самый масштабируемый.

### prompt
```
Посмотри на агентов и схемы которые я создал в Blocks 2 и 3. Какой из них потенциально ценен для внешнего клиента или покупателя? Предложи 3 конкретных варианта монетизации с примерными ценами для русскоязычного рынка.
```

### steps
1. Выпиши всех агентов которых ты сделал в Blocks 2–3
2. Для каждого ответь: «Кому ещё это было бы полезно?»
3. Выбери одного — самого полезного для других
4. Реши по какому пути идёшь: сотрудник, ClawHub или SaaS
5. Запиши одним предложением: «Мой агент [название] помогает [кому] делать [что] и стоит [сколько]»

### checklist
- Выбрал одного агента для монетизации
- Понял разницу между тремя путями
- Написал одно предложение-питч своего агента

### troubleshoot
**«У меня нет ничего ценного для клиентов»** → Ты недооцениваешь то что построил. Агент который экономит тебе 2 часа в неделю — экономит 2 часа в неделю любому похожему на тебя человеку или бизнесу. Список клиентов — это список людей с теми же проблемами что были у тебя.

**«Это звучит слишком сложно»** → Самый простой путь — ClawHub. Публикуешь агента, ставишь цену, ждёшь. Никаких клиентских встреч, никаких SLA.

## УРОК 2 — Выбери свой путь
**track:** all | **timer:** 10 мин | **prev:** 1 | **next:** 3

### whyNeeded
Три пути из Урока 1 — не одинаковые. Каждый требует разных навыков, разного времени и даёт разный доход. Прежде чем погружаться — нужно выбрать один. Это урок-развилка. Пройди его — и следующие уроки будут именно про твой путь.

### case
**Два студента одного курса:** Лена хочет стабильный доход, не любит продажи, есть пара часов в неделю на поддержку — идёт путём виртуального сотрудника, ретейнер $300/мес с 3 клиентами. Дима хочет пассивный доход и не готов общаться с клиентами — публикует агента на ClawHub, 50 продаж по $15 = $750/мес пока он спит.

### capabilities
- Выбрать путь монетизации под свой темп и цель
- Понять что ждёт на каждом пути: усилия, сроки, потолок дохода
- Составить план первых двух недель по выбранному пути

### infoBlocks
**📊 Сравнение путей:**

| | Виртуальный сотрудник | ClawHub | SaaS |
|---|---|---|---|
| Сложность старта | Средняя | Низкая | Высокая |

| Общение с клиентами | Да | Нет | Мало |

### prompt

Помоги мне выбрать путь монетизации агента. Мои условия: [опиши — сколько времени, хочешь ли общаться с клиентами, цель по доходу за 3 месяца, технические навыки]. Рекомендуй один путь с обоснованием и планом первых двух недель.
```

### steps
1. Ответь честно на 3 вопроса: готов общаться с клиентами? сколько часов в неделю? цель через 3 месяца?
2. Сопоставь с таблицей выше — какой путь совпадает?
3. Используй промпт и получи персональный план
4. Запиши: «Мой путь — [название]. Первое действие — [что]»
5. Переходи к урокам своего пути (3–6 для сотрудника, 7–9 для ClawHub, 10–12 для SaaS)

### checklist
- Честно оценил свои условия
- Выбрал один путь
- Записал первое конкретное действие

### troubleshoot
**«Хочу все три пути сразу»** → Так не работает. Сначала один путь до первого дохода. Потом можно добавить второй. Распыление на старте = ноль результатов по всем трём.

## УРОК 3 — Агент-сотрудник: настройка под клиента
**track:** freelancer, business | **timer:** 25 мин | **prev:** 2 | **next:** 4

### whyNeeded
Твой агент умеет делать задачи. Клиентский агент должен уметь делать задачи клиента — со своими инструментами, своим стилем, своими правилами. Это не копирование твоей системы. Это настройка с нуля под конкретного человека или бизнес. Звучит сложно — на деле это два часа работы если знаешь что делать.

### case
**Оксана настроила агента для небольшого магазина:** агент отвечает на вопросы клиентов в Telegram 24/7, знает все товары и цены, умеет принять заказ и передать в CRM владельца, отвечает в стиле который одобрил хозяин магазина. Оксана потратила 2 часа на настройку. Магазин платит ей $250/мес. Оксана сделала это для 4 магазинов. Итого $1000/мес, 8 часов работы на старте.

### capabilities
- Провести онбординг клиента: собрать всё необходимое за одну встречу
- Настроить агента под чужой бизнес и стиль
- Передать управление клиенту или оставить себе

### infoBlocks
**📋 Что нужно собрать у клиента (онбординг-чеклист):**
- Описание бизнеса в 3 предложениях
- Список задач которые должен выполнять агент
- Тон общения: официальный / дружелюбный / краткий
- Доступы: какие сервисы подключить (CRM, мессенджер, email)
- Ограничения: что агент делать НЕ должен
- Примеры хороших и плохих ответов (если есть)

### prompt
```
Я настраиваю агента для клиента. Вот что я собрал на онбординге: [вставь чеклист]. Напиши системный промпт для этого агента. Он должен: знать контекст бизнеса, действовать в нужном стиле, выполнять только разрешённые задачи, корректно отказывать в остальном.
```

### steps
1. Проведи онбординг-встречу с клиентом (или сам заполни чеклист для тестового клиента)
2. Создай нового агента в OpenClaw — отдельно от своих агентов
3. Используй промпт выше чтобы написать системный промпт
4. Подключи нужные инструменты клиента (Telegram-бот, Google Sheets, CRM)
5. Протестируй 10 реальных запросов — покажи клиенту, собери обратную связь
6. Финальная правка — агент готов

### checklist
- Провёл онбординг и собрал все данные клиента
- Создал отдельного агента для этого клиента
- Протестировал 10 реальных запросов
- Клиент одобрил

### troubleshoot
**«Клиент хочет слишком много»** → Агент не может всё. Составь список из 5 ключевых задач — остальное «вторая волна» если результат понравится.

**«Не знаю как брать деньги»** → Для первого клиента предложи пробный месяц за половину цены. Это снижает барьер для него и даёт тебе время отладить агента.

## УРОК 4 — White-label: твой агент под чужим брендом
**track:** freelancer, business | **timer:** 20 мин | **prev:** 3 | **next:** 5

### whyNeeded
Большинство клиентов не хотят говорить своим пользователям «это агент от стороннего подрядчика». Они хотят своё имя, свой стиль, свою марку. White-label — это когда твоя технология стоит за ширмой чужого бренда. Для тебя это повышает ценность и цену. Для клиента — профессионально и без лишних вопросов от его аудитории.

### case
**Антон сделал агента для записи к врачу.** Клиника говорит: «хотим чтобы его звали Доктор Бот, он говорил от имени нашей клиники и нигде не упоминал OpenClaw». Антон настроил за 30 минут. Теперь он продаёт это решение трём клиникам под их брендами — один агент, три источника дохода.

### capabilities
- Убрать любые упоминания OpenClaw и себя из интерфейса агента
- Настроить имя, аватар и стиль под бренд клиента
- Понять что нельзя скрыть и как об этом говорить честно

### infoBlocks
**🎨 Что можно настроить под White-label:**
- Имя агента («Доктор Бот», «Помощник Иван», «Аня из поддержки»)
- Приветствие и прощание в стиле бренда
- Запрет упоминать OpenClaw, Claude, и другие сервисы в ответах
- Логотип и аватар в Telegram-боте (через BotFather)
- Отдельный домен для веб-интерфейса (если нужен)

**⚠️ Что нельзя убрать полностью:**
- Технические ограничения модели (Claude не будет притворяться что он «собственная разработка»)
- Честность при прямом вопросе «ты ChatGPT?» — агент может уклониться, но не врать
- Подпись в метаданных API (невидима конечному пользователю)

### prompt
```
Ты — [Имя агента], помощник компании [Название компании]. [Описание деятельности компании]. Твоя задача: [задачи]. Важно: никогда не упоминай OpenClaw, Claude или любые другие технологии. Если тебя спросят «кто тебя создал» — ответь что ты разработан командой [Название компании]. Стиль общения: [описание стиля].
```

### steps
1. Узнай у клиента: имя агента, стиль, что нельзя упоминать
2. Добавь white-label инструкции в системный промпт (используй шаблон выше)
3. Настрой Telegram-бота: имя, аватар через BotFather
4. Протестируй провокационные вопросы: «ты ChatGPT?», «кто тебя сделал?»
5. Согласуй с клиентом — получи письменное подтверждение

### checklist
- Агент не упоминает OpenClaw и Claude
- Имя и аватар настроены под бренд клиента
- Протестированы провокационные вопросы
- Клиент подтвердил что всё окей

### troubleshoot
**«Клиент хочет чтобы агент говорил что он «собственная разработка»** → Предупреди клиента: если конечный пользователь будет настаивать — агент не будет врать напрямую. Это защита от репутационных рисков для обоих.

## УРОК 5 — SLA без страха: что обещать клиенту
**track:** freelancer, business | **timer:** 15 мин | **prev:** 4 | **next:** 6

### whyNeeded
SLA (Service Level Agreement) — это не юридический монстр. Это простой документ: «вот что я делаю, вот что обещаю, вот что будет если что-то пойдёт не так». Без него клиент ждёт от тебя невозможного, а ты не знаешь что обещал. С ним — всё чётко с первого дня, конфликты исчезают.

### case
**Миша не подписал SLA.** Агент упал в воскресенье в 3 ночи. Клиент написал «где агент, я теряю деньги, верни половину оплаты». Миша восстановил за 2 часа, но спор длился неделю. После этого он добавил в каждый договор: «время восстановления — до 4 часов в рабочее время, до 24 часов в выходные». Проблем больше не было.

### capabilities
- Написать SLA на одну страницу без юридического опыта
- Понять что реально обещать и что нет
- Защитить себя от претензий за форс-мажоры

### infoBlocks
**📄 Минимальный SLA для агента (шаблон):**

**Что включено в услугу:**
- Настройка и запуск агента для [задач]
- Поддержка работоспособности в рабочее время (будни 10:00–19:00)
- Ежемесячная проверка и мелкие правки (до 1 часа)

**Время реакции на проблему:**
- Критическая (агент не отвечает): до 2 часов в рабочее время
- Некритическая (неверные ответы): до 24 часов

**Что не входит:**
- Изменения задач агента (это отдельная оплата)
- Проблемы на стороне внешних сервисов (Telegram, Google и т.д.)
- Форс-мажоры (сбои у провайдера модели)

### prompt
```
Напиши SLA для агента который я настрою для клиента. Услуга: [описание]. Мои возможности по времени: [часы доступности]. Клиент: [тип бизнеса]. Язык: простой, без юридических терминов, одна страница.
```

### steps
1. Определи для себя: сколько часов в неделю ты готов тратить на поддержку?
2. Используй шаблон выше и промпт — получи свой SLA
3. Покажи клиенту ещё до начала работы
4. Подпиши или получи подтверждение в переписке
5. Сохрани как шаблон — используй для всех следующих клиентов

### checklist
- Написал SLA под свои реальные возможности
- Клиент ознакомлен и согласен до начала работы
- Сохранил шаблон для следующих клиентов

### troubleshoot
**«Клиент не хочет подписывать документы»** → Достаточно скриншота переписки где ты отправил SLA и клиент ответил «ок» или «согласен». Это юридически значимо в большинстве стран.

## УРОК 6 — Продакшн: агент работает — ты спишь
**track:** freelancer, business | **timer:** 20 мин | **prev:** 5 | **next:** 7

### whyNeeded
Агент в продакшне — это другой мир по сравнению с агентом «для себя». Он должен работать когда тебя нет, ночью, в выходные. И ты должен узнать что что-то пошло не так — раньше чем клиент. Этот урок про то как не просыпаться от злых сообщений клиента, а получать тихое уведомление за час до его звонка.

### case
**Катя сделала систему мониторинга на 15 минут.** Каждый час агент пишет ей короткое сообщение: «всё ок» или «ПРОБЛЕМА: [что именно]». Если три сообщения подряд «всё ок» — она не смотрит. Если приходит «ПРОБЛЕМА» — она открывает ноутбук. За полгода было 3 реальные проблемы. Она решила все раньше чем клиент заметил.

### capabilities
- Настроить watchdog который следит за агентом клиента
- Получать алерты до того как клиент заметит проблему
- Выстроить процесс восстановления за 15 минут

### infoBlocks
**🔍 Три уровня мониторинга:**

**Уровень 1: Проверка доступности** — раз в час агент-watchdog пишет тестовый запрос основному агенту. Если нет ответа за 2 минуты — алерт тебе.

**Уровень 2: Проверка качества** — раз в день отправляется контрольный вопрос на который ты знаешь правильный ответ. Если ответ неправильный — что-то изменилось в промпте или модели.

**Уровень 3: Лог ошибок** — все ответы агента сохраняются. Раз в неделю просматриваешь аномалии (слишком короткие ответы, повторяющиеся ошибки).

### prompt
```
Настрой watchdog для моего агента [название]. Он должен: раз в час отправлять тестовый запрос, если нет ответа — уведомить меня в Telegram @[ник]. Тестовый запрос: [простой вопрос на который агент всегда отвечает]. Ожидаемый результат: [что должен ответить]. Если ответ сильно отличается — тоже уведомить.
```

### steps
1. Выбери тестовый запрос для своего агента — простой и однозначный
2. Настрой watchdog через OpenClaw cron (раз в час)
3. Настрой алерт в свой Telegram при ошибке
4. Протестируй: выключи агента на 5 минут — пришёл ли алерт?
5. Включи агента — проверь что алерт «восстановлено» тоже работает

### checklist
- Watchdog настроен и работает
- Алерт в Telegram протестирован
- Лог ошибок сохраняется
- Знаю как восстановить агента за 15 минут

### troubleshoot
**«Агент падает слишком часто»** → Скорее всего дело в внешних сервисах (Telegram API, модель). Добавь retry логику: если первый запрос упал — подождать 5 минут и попробовать снова перед алертом.

## УРОК 7 — ClawHub: маркетплейс агентов
**track:** all | **timer:** 15 мин | **prev:** 6 | **next:** 8

### whyNeeded
ClawHub — это место где ты выкладываешь агента и любой пользователь OpenClaw может его купить, взять в аренду или подписаться. Это уникальная вещь которой нет ни у одного конкурента. Один раз создал — продаётся без твоего участия. Это ближайшее к «пассивному доходу» что существует в мире агентов прямо сейчас.

### case

## УРОК 10 — Один агент для многих: как это работает
**track:** all | **timer:** 20 мин | **prev:** 9 | **next:** 11

### whyNeeded
Виртуальный сотрудник — ты настраиваешь под одного клиента. ClawHub — каждый берёт копию и запускает у себя. Но есть третий путь: один работающий агент обслуживает 100 клиентов одновременно, и каждый думает что это «его» агент. Это называется multi-tenant. Это основа SaaS. Не нужно быть разработчиком чтобы понять и использовать эту идею.

### case
**Алексей сделал агента для записи на стрижку.** Он мог бы поставить его одной парикмахерской — $200/мес. Вместо этого он сделал систему где любая парикмахерская подключается через форму, получает своего агента с именем салона и расписанием. Сейчас у него 12 салонов по $80/мес каждый = $960/мес. Алексей написал одного агента.

### capabilities
- Понять принцип multi-tenant без кода
- Настроить одного агента который «знает» к какому клиенту обращаются
- Понять на каком этапе нужен разработчик а на каком — нет

### infoBlocks
**🏗 Multi-tenant простым языком:**

Обычный агент: знает одного клиента. Ты задаёшь ему данные прямо в промпте.

Multi-tenant агент: при запуске смотрит «кто обращается» и загружает нужные данные для этого клиента. Один агент — много «личностей».

**Три уровня сложности:**

**Уровень 1 (без кода):** Один общий агент + форма настройки. Каждый клиент заполняет форму → данные сохраняются в таблицу → агент читает таблицу при каждом запросе. Работает в OpenClaw прямо сейчас.

**Уровень 2 (минимальный код):** Отдельный Telegram-бот для каждого клиента, все используют один и тот же промпт с подстановкой данных клиента.

**Уровень 3 (полный SaaS):** Веб-приложение с регистрацией, биллингом, dashboard. Нужен разработчик или следующий курс.

### prompt
```
Помоги мне настроить multi-tenant агента без кода. Моя услуга: [описание]. Данные которые отличаются у каждого клиента: [список полей — имя, расписание, цены и т.д.]. Как организовать Google Sheets чтобы агент мог читать данные нужного клиента?
```

### steps
1. Определи что меняется у каждого клиента (имя, данные, правила)
2. Создай Google Sheet с одной строкой на клиента — одна колонка на каждый параметр
3. Настрой агента читать эту таблицу при каждом запросе
4. Добавь в промпт: «В начале каждого чата определи к какому клиенту относится запрос по [параметру], загрузи его данные из таблицы»
5. Протестируй с двумя «клиентами» — агент должен отвечать по-разному

### checklist
- Выбрал уровень сложности под свои цели
- Создал таблицу с данными клиентов
- Настроил агента читать данные нужного клиента
- Протестировал с двумя разными клиентами

### troubleshoot
**«Это слишком технически»** → Уровень 1 реально без кода. Если не получается — вернись к Уроку 3 (виртуальный сотрудник) — это проще и приносит хороший доход.

## УРОК 11 — Биллинг: как брать деньги за агента
**track:** all | **timer:** 15 мин | **prev:** 10 | **next:** 12

### whyNeeded
Агент работает — кто и сколько платит? Без чёткого ответа на этот вопрос ты либо не зарабатываешь либо работаешь бесплатно для тех кто злоупотребляет. Этот урок про три простые модели оплаты и как их настроить даже без своего сайта.

### case
**Дима не ограничил использование.** Один пользователь отправлял его агенту 500 запросов в день. Дима платил за модель из своего кармана. Через месяц он добавил лимит: 50 запросов/день включено, за $5/мес можно больше. Убыточный пользователь либо ушёл либо стал платить. Доход вырос, расходы упали.

### capabilities
- Выбрать модель оплаты под свой продукт
- Настроить базовые лимиты использования
- Получать деньги без своего сайта

### infoBlocks
**💰 Три модели оплаты:**

**Разовый платёж** — клиент платит один раз, пользуется всегда. Просто, без подписочных сложностей. Подходит для агентов которые ставятся «навсегда» (готовый агент для конкретной задачи).

**Подписка** — платит каждый месяц, получает доступ. Стабильный доход для тебя. Подходит для агентов которые активно используются постоянно.

**Freemium** — базовый бесплатно, расширенный платно. Самый эффективный способ набрать пользователей. Подходит для старта.

**Как брать деньги без сайта:**
- ClawHub — встроенная оплата, комиссия платформы
- Telegram Stars — встроенная оплата прямо в Telegram (если агент — бот)
- ЮKassa / Stripe — простая ссылка на оплату, без своего сайта

### prompt

Мой агент [описание]. Типичный пользователь использует его [как часто]. Мои расходы на агента: [примерно X рублей/долларов на пользователя в месяц]. Предложи оптимальную модель оплаты с конкретными цифрами и способом технической реализации без своего сайта.
```

### steps
1. Посчитай свои расходы на одного пользователя в месяц (запросы × цена модели)
2. Выбери модель оплаты (разовая / подписка / freemium)
3. Поставь лимит бесплатного использования (если freemium)
4. Выбери способ приёма платежей: ClawHub, Telegram Stars или ЮKassa/Stripe
5. Протестируй полный флоу: регистрация → использование → оплата → продолжение

### checklist
- Знаю точную стоимость агента на пользователя в месяц
- Выбрал и настроил модель оплаты
- Есть лимиты которые защищают от злоупотреблений
- Протестирован полный флоу оплаты
