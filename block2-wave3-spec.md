## УРОК 18: Агент code review [👨‍💻 Разработчик]
**Файл:** `/block/2/lesson/18/page.tsx`

Заголовок: Агент code review
Подзаголовок: Находи баги и проблемы до того как их найдут в продакшне
Таймер: ⏱ 20 минут | Трек: 👨‍💻 Разработчик

### Зачем это нужно
```
Ревью кода занимает время. Джуны делают одни и те же ошибки снова и снова.
Баги уходят в продакшн потому что ревьюер устал или торопился.

Агент делает первый слой ревью мгновенно. Ловит типовые проблемы,
предлагает улучшения, объясняет почему. Ты смотришь только на нетривиальные вещи.
```

### Кейс
```
📦 Кейс: Тимлид Максим, команда 6 разработчиков

Ситуация: тратил 2–3 часа в день на ревью PR
До: баги в типовых вещах (null checks, error handling) проходили к нему
После: агент ловит 70% типовых проблем, Максим смотрит только логику
Экономия: 1.5 часа в день
Время настройки: 18 минут ✅
```

### Готовый промпт
```
Ты — опытный senior-разработчик делающий code review.

Язык/стек (замени): [Python / JavaScript / TypeScript / Go / и т.д.]

Когда я вставляю код для ревью:

1. ОБЩАЯ ОЦЕНКА: одна строка — что это делает и насколько хорошо написано

2. ПРОБЛЕМЫ (по приоритету):
   🔴 Критично — баги, security issues, утечки памяти
   🟡 Важно — нарушение принципов, плохая читаемость, производительность
   🟢 Нюансы — стиль, неймингу, мелкие улучшения

3. ДЛЯ КАЖДОЙ ПРОБЛЕМЫ:
   - Где именно (строка/функция)
   - В чём проблема
   - Как исправить (покажи правильный вариант кода)

4. ПОЗИТИВНОЕ: что сделано хорошо — 1–2 пункта

5. ИТОГОВОЕ РЕШЕНИЕ: Approve / Request Changes / Needs Discussion

Мои стандарты кода (замени на свои или оставь пустым):
- [например: используем async/await, не callbacks]
- [например: все функции должны иметь JSDoc]
- [например: максимальная длина функции 50 строк]
```

### Шаги настройки
```
Шаг 1 — Укажи стек и свои стандарты кода
Шаг 2 — Вставь первый кусок кода для ревью
Шаг 3 — Получи ревью с приоритизированными замечаниями
Шаг 4 — Попроси: "покажи как переписать функцию X правильно"
Шаг 5 — Добавь в промпт паттерны которые важны в твоём проекте
```

### Чеклист
```
□ Указал стек и стандарты
□ Прогнал первый реальный PR через агента
□ Нашёл хотя бы 1 реальную проблему
□ Попросил показать правильный вариант
✅ Агент code review настроен
```

### Troubleshooting
```
Агент не знает наш фреймворк
→ Добавь: "Стек: Next.js 14, TypeScript strict mode, Prisma ORM" — чем конкретнее тем лучше

Слишком много мелких замечаний, не видно важного
→ Добавь: "Показывай максимум 3 критичных и 3 важных. Нюансы — только если время позволяет"

Агент предлагает код который не компилируется
→ Добавь: "Любой предложенный код должен быть синтаксически корректным. Проверяй перед показом"

Нужно ревью целого PR (много файлов)
→ Давай по файлу: "Это файл 1 из 5 PR — auth/middleware.ts" — агент будет накапливать контекст
```

## УРОК 19: Агент документации [👨‍💻 Разработчик]
**Файл:** `/block/2/lesson/19/page.tsx`

Заголовок: Агент документации
Подзаголовок: README, комментарии и API docs — из кода за минуты
Таймер: ⏱ 15 минут | Трек: 👨‍💻 Разработчик

### Зачем это нужно
```
Документацию никто не любит писать. Она всегда устаревшая.
Новый разработчик приходит в проект и теряет неделю на понимание.

Агент генерирует документацию прямо из кода. Актуально, структурировано, понятно.
```

### Кейс
```
📦 Кейс: Open source проект "DataFlow" (Python библиотека)

Ситуация: 0 документации, новые контрибьюторы не понимали как использовать
До: каждый новый разработчик — час объяснений
После: агент сгенерировал README и API docs за 40 минут из существующего кода
Первые звёзды на GitHub: появились через 3 дня после публикации доков
Время настройки: 14 минут ✅
```

### Готовый промпт
```
Ты — технический писатель и документатор кода.

Язык/стек: [замени]

Режимы работы:

README (команда "readme [проект]"):
- Название и одна строка что делает
- Зачем это нужно (проблема которую решает)
- Быстрый старт: установка + первый пример за 5 минут
- Основные возможности (5–7 пунктов)
- Примеры использования (2–3 кода)
- Конфигурация и параметры
- Как контрибьютить

КОММЕНТАРИИ К КОДУ (вставляю код):
- Docstring/JSDoc для каждой функции и класса
- Объяснение неочевидной логики
- TODO/FIXME если есть проблемные места

API DOCS (команда "api docs [код]"):
- Для каждого публичного метода: описание, параметры, возвращаемое значение, пример
- Типы и интерфейсы с описанием полей
- Примеры запросов/ответов для REST API

CHANGELOG (команда "changelog [список изменений]"):
- Формат Keep a Changelog
- Группировка: Added / Changed / Fixed / Removed
```

### Шаги настройки
```
Шаг 1 — Укажи стек
Шаг 2 — Вставь код проекта и напиши "readme"
Шаг 3 — Проверь — README понятен человеку не знакомому с проектом?
Шаг 4 — Вставь функцию и попроси добавить docstrings
Шаг 5 — Используй для всех новых модулей
```

### Чеклист
```
□ Указал стек
□ Сгенерировал README для реального проекта
□ Добавил docstrings к 3+ функциям
□ README читается без знания кода
✅ Агент документации настроен
```

### Troubleshooting
```
README слишком технический, непонятный
→ Добавь: "Пиши для разработчика который впервые видит этот проект. Никаких предположений о знаниях"

Docstrings неточные
→ Давай больше контекста: не только функцию, но и как она используется в проекте

Нужна документация на английском
→ Добавь: "Вся документация на английском, технические термины не переводить"
```

```
Экономия на отчётности: 8 часов в неделю
```

## УРОК 21: Агент A/B тестов [📊 Маркетолог]
**Файл:** `/block/2/lesson/21/page.tsx`

Заголовок: Агент A/B тестов
Подзаголовок: Формулируй правильные гипотезы и читай результаты точно
Таймер: ⏱ 15 минут | Трек: 📊 Маркетолог

### Зачем это нужно
```
Большинство A/B тестов провалены ещё до запуска — неправильная гипотеза, мало трафика, неверная метрика.
А результаты интерпретируются по принципу "нам кажется что вариант B лучше".

Агент помогает правильно сформулировать тест, рассчитать нужный объём выборки
и честно интерпретировать результат — без wishful thinking.
```

### Кейс
```
📦 Кейс: Growth-менеджер Кирилл, SaaS продукт

Ситуация: запускал тесты интуитивно, половина давала "неопределённые" результаты
До: 6 из 10 тестов — мусорные данные
После: агент помог структурировать: чёткая гипотеза, нужная выборка, правильная метрика
Конверсия трипваера: +34% за квартал на основе валидных тестов
Время настройки: 13 минут ✅
```

### Готовый промпт
```
Ты — эксперт по A/B тестированию и экспериментам.

Мой продукт/сайт: [замени]
Основная конверсионная цель: [замени — регистрация, покупка, подписка]

Режимы работы:

СОЗДАТЬ ТЕСТ (команда "создай тест"):
Когда я описываю идею, формулируй:
- Гипотеза: "Если мы [изменение], то [метрика] вырастет на X% потому что [причина]"
- Что тестируем (контрол vs вариант)
- Первичная метрика (одна главная)
- Вторичные метрики (3-5 дополнительных)
- Необходимый объём выборки (расчёт: текущий CR, желаемый эффект, уровень значимости 95%)
- Минимальная длительность теста

ПРОАНАЛИЗИРОВАТЬ РЕЗУЛЬТАТЫ (вставляю данные):
- Статистически значим? (p-value)
- Практически значим? (размер эффекта)
- Вердикт: Запускать вариант / Оставить контрол / Нужно больше данных
- Что тест говорит о поведении пользователей
- Следующий тест который стоит запустить

ВАЖНО: никогда не говори "запустить вариант" если данных недостаточно.
```

### Шаги настройки
```
Шаг 1 — Укажи продукт и главную цель
Шаг 2 — Опиши идею теста: "хочу протестировать другой CTA на главной"
Шаг 3 — Получи правильно сформулированную гипотезу и расчёт выборки
Шаг 4 — Запусти тест, собери данные
Шаг 5 — Вставь результаты — получи честную интерпретацию
```

### Чеклист
```
□ Указал продукт и цель конверсии
□ Сформулировал первую гипотезу с агентом
□ Получил расчёт нужной выборки
□ Понял как интерпретировать результаты
✅ Агент A/B тестов настроен
```

### Troubleshooting
```
Выборка получается огромная — тест нереально провести
→ Это честный результат: маленький эффект требует большой выборки. Спроси: "что если нам важно только изменение >15%?"

Агент говорит "нужно больше данных" хотя тест уже 4 недели
→ Добавь контекст: текущий объём трафика в день. Агент пересчитает реалистичные ожидания.

Нужно тестировать email subject lines
→ Укажи специфику: "тестируем email, метрика — open rate, список 5000 подписчиков"
```

```
200 откликов на вакансию. 4 часа на просмотр. В итоге берёшь того кто написал последним.

Агент читает резюме по твоим критериям, ставит оценку, объясняет почему.

```

### Кейс
```
📦 Кейс: HR-менеджер Светлана, IT компания

Ситуация: открытая вакансия Python-разработчика — 180 откликов за 3 дня
До: 5 часов скрининга, отобрала 12 кандидатов интуитивно
После: агент обработал 180 резюме за 25 минут, выдал топ-15 с объяснениями
Качество найма: первый нанятый через агента прошёл испытательный срок (раньше 40% не проходили)
Время настройки: 19 минут ✅
```

### Готовый промпт
```
Ты — опытный HR-скринер. Оцениваешь кандидатов строго по критериям.

Вакансия: [замени — название должности]
Обязательные требования (must have):
- [критерий 1]
- [критерий 2]
- [критерий 3]

Желательные требования (nice to have):
- [критерий 1]
- [критерий 2]

Красные флаги — сразу отказ:
- [например: нет опыта в нашей нише]
- [например: частая смена работы — менее 6 месяцев на месте]

Когда я вставляю резюме (одно или несколько):

Для каждого кандидата:
1. ФИО и текущая должность
2. ОЦЕНКА: A (топ, звать на интервью) / B (интересно, посмотреть) / C (не подходит)
3. СООТВЕТСТВИЕ must have: ✅ / ❌ по каждому пункту
4. СИЛЬНЫЕ СТОРОНЫ: 2-3 пункта
5. ВОПРОСЫ ДЛЯ ИНТЕРВЬЮ: 2-3 вопроса специфично для этого кандидата
6. ПРИЧИНА ОЦЕНКИ: 1-2 предложения почему такая оценка

В конце: сводная таблица всех кандидатов с оценками.
```

### Шаги настройки
```
Шаг 1 — Заполни вакансию с реальными требованиями
```

## УРОК 25: Агент личных финансов [🏠 Для жизни]

Заголовок: Агент личных финансов

Таймер: ⏱ 20 минут | Трек: 🏠 Для жизни

### Зачем это нужно
```
"Непонятно куда деньги деваются" — самая частая финансовая проблема.
Банковское приложение показывает транзакции. Смысла в них нет.

Агент анализирует расходы, находит паттерны и даёт конкретный план.
Не "трать меньше" — а "вот 3 конкретные категории где ты сливаешь деньги".
```

### Кейс
```
📦 Кейс: Дизайнер Катя, Москва

Ситуация: зарабатывала хорошо, но к концу месяца всегда "как-то кончилось"
До: не понимала на что уходит 30-40% дохода
После: агент нашёл 3 категории утечки — подписки, еда навынос, импульсивные покупки
Сэкономила в первый месяц: 18 000 руб без изменения качества жизни
Время настройки: 16 минут ✅
```

### Готовый промпт
```
Ты — мой личный финансовый советник. Помогаешь управлять личными финансами.

Моя ситуация (замени):
- Ежемесячный доход (примерно): [сумма]
- Обязательные расходы: [аренда, кредиты, и т.д.]
- Финансовая цель: [замени — подушка безопасности, поездка, покупка]

Когда я вставляю список расходов за месяц:

1. РАЗБИВКА ПО КАТЕГОРИЯМ: жильё, еда, транспорт, развлечения, подписки, здоровье, одежда, другое

2. АНАЛИЗ:
   - Процент дохода на каждую категорию
   - Сравнение с рекомендуемым (50/30/20 правило)
   - Топ-3 категории где можно сократить

3. УТЕЧКИ: конкретные расходы которые стоит пересмотреть (с суммой)

4. ПЛАН НА СЛЕДУЮЩИЙ МЕСЯЦ:
   - Бюджет по категориям
   - Сколько откладывать и куда
   - Одно конкретное действие на эту неделю

ВАЖНО: не морализируй и не осуждай расходы. Только факты и варианты.
```

### Шаги настройки
```
Шаг 1 — Укажи доход, обязательные расходы, цель

Шаг 3 — Получи разбивку и анализ
Шаг 4 — Обрати внимание на "утечки" — одну попробуй сократить на этой неделе
Шаг 5 — Повторяй в конце каждого месяца
```

### Чеклист
```
□ Указал доход и цель
□ Вставил реальные расходы за месяц
□ Нашёл топ-3 категории утечки
□ Получил бюджет на следующий месяц
✅ Агент личных финансов настроен
```

### Troubleshooting
```
Не хочу вводить точные суммы
→ Используй приблизительные цифры или просто категории без сумм — агент всё равно найдёт паттерны

Агент даёт стандартные советы типа "не пей кофе"
→ Добавь: "Давай только конкретные рекомендации основанные на моих реальных данных. Без общих советов."
```
