## УРОК 1 — Один агент — это потолок
**track:** all | **timer:** 15 мин | **prev:** null | **next:** 2

### whyNeeded
Ты уже умеешь запускать агента для реальных задач. Теперь вопрос: а что когда задача сложнее? Когда нужно одновременно ресёрчить, писать, проверять и публиковать? Один агент с этим не справится — не потому что он плохой, а потому что у него физические ограничения. Этот урок объясняет где потолок — и почему пора его пробить.

### case
**Маша ведёт Telegram-канал.** Каждое утро она вручную просит агента найти новости, потом отдельно — написать пост, потом — проверить факты, потом — опубликовать. Это 4 отдельных запроса, 4 ожидания, 4 копирования результата вручную. Мультиагент сделал бы всё это за неё пока она пьёт кофе.

### capabilities
- Понять где один агент физически не справляется
- Увидеть задачу которую стоит автоматизировать в своём треке
- Получить первый набросок мультиагентной цепочки

### prompt
```
Покажи мне какие задачи я сейчас делаю в несколько шагов — и как их объединить в одну автоматизированную цепочку агентов. Мои типичные задачи: [ОПИШИ СВОИ ЗАДАЧИ]
```

### steps
1. Вспомни любую задачу из Block 2 где ты делал несколько запросов подряд
2. Запиши: Шаг 1 → Шаг 2 → Шаг 3 (что делал каждый раз)
3. Теперь представь каждый шаг — это отдельный агент-специалист
4. Спроси своего агента: «Как бы ты разделил задачу X между тремя агентами?»
5. Получи первый набросок своей будущей мультиагентной системы

### checklist
- Нашёл задачу где делаешь 3+ шага вручную
- Понял разницу: один агент vs команда агентов
- Вижу конкретную задачу для автоматизации в своём треке

### troubleshoot
**«Мой агент справляется с задачами и так»** → Справляется сейчас, пока задач мало. Когда их станет больше — этот урок предотвращает будущие проблемы.

**«Я не понимаю зачем усложнять»** → Цель не усложнить, а убрать себя из процесса. Ты сейчас — часть цепочки. Мультиагент делает тебя ненужным в рутине.

## УРОК 2 — Роли в команде агентов
**track:** all | **timer:** 20 мин | **prev:** 1 | **next:** 3

### whyNeeded
Прежде чем строить команду — нужно знать кто в ней может быть. Как в настоящей компании: есть менеджер, есть исполнители, есть контролёр качества. У агентов всё то же самое — и это не метафора, а буквальная архитектура которую ты сейчас настроишь.

### case
**Твой Orchestrator получает задачу** «напиши пост про новость дня». Он отдаёт Researcher'у: «найди главную крипто-новость». Researcher возвращает факты. Orchestrator отдаёт Writer'у: «напиши пост на основе этих фактов». Writer возвращает текст. Monitor проверяет: «нет ли в тексте ошибок и неточностей». Ты получаешь готовый пост. Весь процесс — без твоего участия.

### capabilities
- Знать три ключевые роли: Orchestrator, Worker, Monitor
- Выбрать нужные роли для своей задачи
- Получить разбивку задачи на подзадачи от агента

### infoBlocks
**🎯 Orchestrator (Управляющий)** — принимает задачу от тебя, разбивает на подзадачи, раздаёт исполнителям, собирает результат. Ты общаешься только с ним.

**⚙️ Workers (Исполнители)** — специализированные агенты: Researcher (ищет инфу), Writer (пишет), Coder (делает код), Publisher (публикует). Каждый знает только своё.

**👁️ Monitor (Контролёр)** — следит чтобы всё работало. Если Worker завис — поднимает тревогу. Проверяет качество до отправки результата тебе.

### prompt
```
Ты — Orchestrator. Твоя задача: [ЗАДАЧА]. Разбей её на подзадачи для следующих исполнителей: Researcher, Writer, Monitor. Для каждого укажи точно что он должен сделать и что вернуть.
```

### steps
1. Выбери одну задачу из своего трека которую хочешь автоматизировать
2. Определи нужные роли: Orchestrator нужен всегда. Какие Workers?
3. Напиши для каждого Worker'а одно предложение: «Твоя задача — ...»
4. Используй промпт выше — проверь как Orchestrator разбивает задачу
5. Получи план своей первой мультиагентной системы

### checklist
- Понял разницу между Orchestrator и Worker
- Выбрал роли для своей задачи
- Получил разбивку задачи на подзадачи от агента

### troubleshoot
**«Сколько нужно агентов?»** → Начни с трёх: Orchestrator + 1–2 Worker. Monitor добавишь потом. Чем проще — тем надёжнее.

**«Можно ли одному агенту играть несколько ролей?»** → Можно, но нежелательно. Специализация = качество + предсказуемость.

## УРОК 4 — Делегирование и изоляция
**track:** all | **timer:** 20 мин | **prev:** 3 | **next:** 5

### whyNeeded
Ты уже знаешь кто какую роль играет. Теперь главное правило мультиагентных систем: агент не должен знать больше чем ему нужно для своей задачи. Это не ограничение — это защита от ошибок, утечек и хаоса. Правильное делегирование = чёткий Input + чёткий Output для каждого.

### case
**Сергей запустил систему без изоляции** — все агенты видели весь контекст разговора. Researcher случайно «помог» Writer'у и добавил факты прямо в черновик поста. Writer решил что это уже готовый текст и не писал сам. На выходе — каша. После изоляции: каждый агент получает только свой файл задачи. Качество выросло в 3 раза.

### capabilities
- Определить Input и Output для каждого агента системы
- Убрать лишний контекст из задач агентов
- Протестировать изолированную задачу для одного Worker'а

### infoBlocks
**🎯 Минимум контекста** — агент получает только то что нужно для своей задачи. Researcher не знает что будет с его данными. Writer не знает откуда пришли факты.

**📦 Изолированная память** — каждый агент работает со своим «файлом задачи». Нет общей памяти которую все могут испортить.

**📋 Чёткий формат вывода** — каждый Worker возвращает результат в определённом формате. Orchestrator знает чего ждать.

### prompt
```
Я передаю тебе задачу как Worker-[РОЛЬ]. Вот твоё задание: [ЗАДАНИЕ]. Вот входные данные: [ДАННЫЕ]. Верни результат в формате: [ФОРМАТ]. Больше ничего не добавляй и не интерпретируй — только выполни своё задание.
```

### steps
1. Возьми систему из урока 2 (твои роли и задачи)
2. Для каждого Worker'а напиши: что он получает (Input) и что возвращает (Output)
3. Проверь: знает ли каждый Worker что-то лишнее? Если да — убери
4. Протестируй: используй промпт выше для одного Worker'а
5. Убедись что формат вывода именно такой какой ты ожидал

### checklist
- Определил Input и Output для каждого агента в системе
- Убедился что никто не знает лишнего
- Протестировал изолированную задачу для одного Worker'а

### troubleshoot
**«Агент игнорирует формат вывода»** → Добавь в конце промпта: «Твой ответ должен содержать ТОЛЬКО следующие поля: [список]. Без вступлений и объяснений.»

**«Данные не передаются правильно»** → Используй структурированный формат: `### INPUT\n[данные]\n### END INPUT`

## УРОК 5 — Cron + мультиагент: система работает без тебя
**track:** all | **timer:** 25 мин | **prev:** 4 | **next:** 6

### whyNeeded
Это урок где слоган блока становится реальностью. «Перестань управлять агентами — пусть они управляют собой.» Cron-расписание превращает твою мультиагентную систему из инструмента в автономный механизм. Ты настраиваешь один раз — система работает каждый день.

### case
**Дима настроил утренний пайплайн:** в 7:00 Researcher собирает новости → в 7:15 Writer пишет дайджест → в 7:30 система отправляет готовый текст ему в Telegram. Дима просыпается в 8:00 и читает уже готовый дайджест. Агенты отработали пока он спал.

### capabilities
- Настроить последовательный запуск агентов по расписанию
- Использовать файловое промежуточное хранилище между агентами
- Настроить уведомление о результате или ошибке

### infoBlocks
**⏰ Последовательный запуск** — агент 2 запускается только после того как агент 1 вернул результат.

**📁 Файловое хранилище** — агент 1 пишет результат в файл → агент 2 читает из файла. Надёжно и без конфликтов.

**🔔 Уведомление в конце** — Orchestrator сообщает тебе что всё готово (или что что-то пошло не так).

**🛡️ Таймаут** — если агент не ответил за N минут → система помечает задачу как failed и уведомляет.

### prompt
```
Настрой расписание для следующей цепочки агентов: [ЦЕПОЧКА]. Первый агент запускается в [ВРЕМЯ]. Каждый следующий — через [ИНТЕРВАЛ] после предыдущего. В конце — уведомление мне с итогом или описанием ошибки.
```

### steps
1. Выбери задачу из своего трека которую хочешь запускать автоматически
2. Разбей на шаги: что делает агент 1, агент 2, агент 3 — и в какое время
3. Настрой cron для первого агента через OpenClaw
4. Добавь уведомление: «пришли мне результат когда всё готово»
5. Проверь что цепочка срабатывает правильно

### checklist
- Задача разбита на временную цепочку агентов
- Cron настроен хотя бы для первого агента
- Уведомление о результате приходит тебе

### troubleshoot
**«Агент 2 запускается раньше чем агент 1 закончил»** → Добавь буфер времени между запусками или используй файловую зависимость: агент 2 проверяет что файл от агента 1 существует.

**«Cron не срабатывает»** → Проверь формат расписания. Каждый день в 7:00 → `0 7 * * *`

## УРОК 6 — Агент-проверяльщик (QA-агент)
**track:** all | **timer:** 20 мин | **prev:** 5 | **next:** 7

### whyNeeded
Агенты ошибаются. Иногда Writer придумывает факты. Иногда Researcher пропускает ключевую информацию. QA-агент — это твой контролёр качества который читает результат до тебя и ловит проблемы. Без него ты либо проверяешь всё сам, либо получаешь брак.

### case
**Катя запустила контент-систему без QA.** Через неделю обнаружила что Writer написал пост со статистикой за 2021 год — Researcher принёс устаревшие данные. 500 человек уже прочитали. После добавления QA-агента: он проверяет каждый факт, дату и цифру. Если что-то не так — блокирует публикацию и сообщает что именно нужно исправить.

### capabilities
- Написать промпт для QA-агента под свою задачу
- Добавить QA-шаг в свою мультиагентную систему
- Настроить что происходит при APPROVED и при замечаниях

### infoBlocks
**✅ Что проверяет QA-агент:** фактология и актуальность данных, тон и соответствие стилю, полнота ответа на исходную задачу, длина и формат под платформу.

**❌ Что возвращает при проблемах:** конкретный список замечаний с указанием что именно не так — не просто «плохо», а «в абзаце 2 цифра устарела, актуальная — X».

### prompt
```
Ты — QA-агент. Прочитай следующий результат работы и проверь: 1) Все факты и цифры актуальные и точные? 2) Тон и стиль соответствует: [ОПИСАНИЕ СТИЛЯ]? 3) Задача «[ИСХОДНАЯ ЗАДАЧА]» выполнена полностью? Если всё ок — ответь одним словом: APPROVED. Если есть проблемы — перечисли конкретные замечания с указанием что именно не так.
```

### steps
1. Возьми любой результат от своего агента из Block 2
2. Запусти промпт QA-агента — посмотри что он найдёт
3. Добавь QA-шаг в свою мультиагентную систему из предыдущих уроков
4. Настрой: если QA вернул APPROVED → идём дальше. Если замечания → пауза и уведомление тебе
5. Протестируй намеренно «плохой» результат — убедись что QA его поймает

### checklist
- Понял что проверяет QA-агент
- Протестировал QA-промпт на реальном тексте
- Добавил QA в свою систему и настроил ветвление

### troubleshoot
**«QA одобряет всё подряд»** → Сделай критерии конкретнее: «Если любая цифра старше 2024 года — это ошибка. Если нет призыва к действию — это замечание.»

**«QA находит слишком много замечаний на хорошем тексте»** → Уточни допуски: «Это разговорный текст, не академическая статья. Допустимо: [список].»

## УРОК 7 — Живой пример: система Zen + Dev + Mark
**track:** all | **timer:** 25 мин | **prev:** 6 | **next:** по треку (8/10/12/14)

### whyNeeded
Теория хорошо — но лучше один раз увидеть работающую систему. Вот реальная мультиагентная команда которая работает каждый день: Zen (управляющий), Dev (разработчик), Mark (контент), Search (ресёрч), Brow (автоматизация). Разберём как они взаимодействуют — и найдём аналогии для твоей системы.

### case
**Задача «сделай пост о новом продукте»** попадает к Zen → он просит Search найти данные о продукте → Search возвращает файл с фактами → Zen просит Mark написать пост на основе фактов → Mark возвращает текст → Zen проверяет качество → отправляет хозяину на апрув → после апрува Brow публикует. Ни один агент не делал чужую работу. Хозяин увидел только финальный результат.

### capabilities
- Понять как реальная команда агентов взаимодействует
- Получить схему своей системы на основе разбора
- Найти аналогии с системой Zen+Dev+Mark для своего трека

### infoBlocks
**🧠 Zen (Orchestrator)** — принимает задачи, делегирует, контролирует результат, общается с хозяином системы.

**👨‍💻 Dev (Worker: код)** — только технические задачи: создание и правка кода, деплой. Ничего кроме кода.

**📣 Mark (Worker: контент)** — только контент: посты, тексты, социальные сети. Ничего кроме контента.

**🔍 Search (Worker: ресёрч)** — только сбор данных: поиск, анализ, структурированные отчёты.

**🤖 Brow (Worker: браузер)** — только веб-автоматизация: заполнение форм, клики, публикации.

**Правила системы:** никто не общается с хозяином кроме Zen. Каждый агент работает только в своей области. Результат → файл → следующий агент. Ошибки → Zen собирает и докладывает.

### prompt
```
Опиши мне структуру мультиагентной системы для задачи: [ЗАДАЧА]. Укажи: 1) Список агентов и их роли, 2) Порядок работы шаг за шагом, 3) Что каждый агент получает и что возвращает, 4) Кто общается с пользователем.
```

### steps
1. Прочитай описание системы Zen+Dev+Mark+Search+Brow выше
2. Найди аналогии со своей задачей: какие роли тебе нужны?
3. Используй промпт — получи схему своей системы
4. Сравни с системой выше: что похоже, что отличается?
5. Сохрани схему — она станет основой для твоего капстоуна в уроке 20

### checklist
- Понял как взаимодействуют агенты в реальной системе
- Получил схему своей системы от агента
- Нашёл аналогии со своим треком

### troubleshoot
**«Моя задача проще — мне не нужно 5 агентов»** → Отлично. Начни с 2–3. Систему можно наращивать. Главное — правильная архитектура с начала.

**«Не понимаю как агенты передают данные друг другу»** → В следующих уроках разберём на практике твоего трека.

## УРОК 8 — Фрилансер: три агента которые заменяют менеджера
**track:** freelancer | **timer:** 30 мин | **prev:** 7 | **next:** 9

### whyNeeded
Фрилансер тратит 30–40% времени не на работу, а на управление работой: переписка с клиентами, составление КП, трекинг задач и дедлайнов. Система трёх агентов берёт это на себя. Ты занимаешься только тем за что платят.

### case
**Павел раньше тратил 2 часа в день** на переписку и отчёты. После запуска системы: агент-КП делает предложение за 3 минуты, агент-трекер сам напоминает о дедлайнах, агент-коммуникатор отвечает клиентам пока Павел работает. 2 часа вернулись — это 40+ часов в месяц.

### capabilities

### capabilities

**📤 Publisher** — получает готовый текст, адаптирует под каждую платформу (Telegram — одно, Instagram — другое), публикует или ставит в очередь.

### prompt
```
Запускай ежедневный контент-конвейер в [ВРЕМЯ]. Шаг 1 — Researcher: собери главные новости в нише [НИША] за последние 24 часа. Шаг 2 — Writer: на основе дайджеста напиши пост в стиле [ОПИСАНИЕ СТИЛЯ]. Шаг 3 — Publisher: адаптируй пост под [ПЛАТФОРМЫ]. Пришли мне финальные варианты на проверку.
```

### steps
1. Определи свою нишу и опиши стиль своего контента
2. Настрой Researcher — источники которые он должен мониторить
3. Настрой Writer с 3–5 образцами твоих лучших постов как примером стиля

5. Запусти конвейер вручную → оцени результат → настрой cron

### checklist
- Researcher возвращает структурированный дайджест
- Writer пишет в твоём стиле (не generic)
- Publisher адаптирует под нужные платформы
- Конвейер запускается по расписанию

**«Researcher берёт старые новости»** → Добавь в промпт: «Используй только материалы опубликованные за последние 24 часа. Сегодня: [дата].»

**Анастасия занимается криптой и стартапами.** Каждое воскресенье агент-планировщик присылает: «3 возможности недели: 1) [Проект X] — IDO через 5 дней, community активна, показатели выше среднего. 2) [Токен Y] — RSI 28 (oversold), исторически возвращается к среднему за 2 недели. 3) [Стратегия Z] — арбитраж между CEX/DEX, спред 2.3%.» Анастасия анализирует и принимает решения — агент нашёл, она решает.

### capabilities
- Настроить еженедельный поиск возможностей в своей нише
- Получать конкретные actionable возможности (не общие советы)
- Автоматизировать доставку отчёта по расписанию

### infoBlocks
**🎯 Недооценённые активы** — по твоим критериям (P/E, RSI, объёмы).

**📊 Арбитраж** — разница цен между платформами или форматами.

**📈 Зарождающиеся тренды** — что набирает аудиторию но ещё не mainstream.

**💼 Новые инструменты** — сервисы, партнёрства, airdrop'ы в твоей нише.

### prompt
```
Ты — агент поиска возможностей в нише [НИША]. Каждую неделю находи 3 возможности достойные внимания. Критерии: [КРИТЕРИИ]. Формат отчёта: 📌 [название] | Почему сейчас: [причина] | Риск: [низкий/средний/высокий] | Действие: [что сделать] | Дедлайн: [если есть]
```

### steps
1. Опиши свою нишу и критерии хорошей возможности
2. Настрой агента с промптом выше
3. Запроси первый отчёт — «найди 3 возможности прямо сейчас»
4. Оцени качество: релевантны ли возможности твоим критериям?
5. Настрой cron: каждое воскресенье в 9:00 — еженедельный отчёт

### checklist
- Агент понимает твою нишу и критерии
- Первый отчёт содержит конкретные релевантные возможности
- Еженедельный cron настроен

### troubleshoot
**«Возможности слишком общие»** → Добавь в промпт: «Каждая возможность должна включать конкретное действие которое я могу сделать сегодня.»

**«Агент повторяет одни и те же возможности»** → Добавь: «Не повторяй возможности из предыдущих отчётов. Ищи что-то новое каждую неделю.»

## УРОК 16 — Токены, бюджеты, watchdog
**track:** all | **timer:** 20 мин | **prev:** 9/11/13/15 | **next:** 17

### whyNeeded
Мультиагентная система работает в фоне. Если что-то пойдёт не так — агент может зациклиться и потратить деньги пока ты спишь. Watchdog — это агент-надзиратель который следит за остальными агентами и останавливает систему при первых признаках проблемы.

### case
**Олег запустил ночной пайплайн сбора данных.** Один из агентов попал на страницу с защитой Cloudflare и начал бесконечно пытаться получить доступ. За ночь — 2000+ запросов и деньги на ветер. Watchdog поймал бы это через 15 минут: «агент зациклился на URL X — останавливаю и уведомляю.»

### capabilities
- Определить лимиты для своей системы
- Настроить watchdog с таймаутами и бюджетами
- Получать heartbeat о состоянии системы

### infoBlocks
**💰 Бюджет токенов** — устанавливаешь лимит на задачу или на день. При превышении — стоп и уведомление.

**⏰ Таймауты** — если агент не ответил за N минут — считается зависшим, задача помечается failed.

**🔄 Зацикливание** — агент повторяет одно и то же более 3 раз — признак бесконечного цикла.

**📊 Heartbeat** — каждые 10 минут агент сигнализирует что он жив и что делает.

### prompt
```
Ты — watchdog-агент. Следи за системой [ОПИСАНИЕ]. Останавливай и уведомляй меня если: 1) Агент не отвечает более [N] минут. 2) Агент повторил одну задачу более 3 раз. 3) Потрачено токенов более [ЛИМИТ] за сессию. Каждые 10 минут присылай heartbeat: «✅ Система работает | [что делает каждый агент]»
```

### steps
1. Определи лимиты для своей системы: таймаут, бюджет токенов
2. Настрой watchdog с промптом выше
3. Добавь heartbeat — краткий статус каждые 10 минут
4. Протестируй: запусти систему и посмотри на первые heartbeat сообщения
5. Настрой действие при срабатывании: пауза системы + уведомление тебе

### checklist
- Лимиты определены и прописаны в watchdog
- Heartbeat приходит регулярно
- Понял что делать если watchdog поймал проблему

### troubleshoot
**«Watchdog сам потребляет много ресурсов»** → Сделай watchdog максимально простым — только мониторинг и уведомления. Никаких аналитических задач.

**«Слишком много heartbeat сообщений»** → Измени интервал на 30 минут или: «Присылай heartbeat только если что-то изменилось.»

## УРОК 17 — Обработка ошибок и fallback
**track:** all | **timer:** 20 мин | **prev:** 16 | **next:** 18

### whyNeeded
В любой системе что-то идёт не так: API не отвечает, данные пришли в неожиданном формате, агент вернул пустой результат. Fallback-стратегии — это заранее прописанные планы Б, В и Г. Система которая умеет восстанавливаться — надёжнее системы которая «никогда не ломается» (такой просто не существует).

### case
**Система Владимира получала данные о погоде** для планировщика. API лёг на техобслуживание. Без fallback — система упала и ничего не сделала. С fallback: попытка 1 (retry) → попытка 2 (retry) → переключение на резервный API → если и он недоступен → планировщик работает без погоды и уведомляет Владимира что данные неполные. Система не упала.

### capabilities
- Определить критические точки своей системы где может быть сбой
- Написать трёхуровневый fallback для каждой точки
- Протестировать восстановление на намеренной ошибке

### infoBlocks
**🟡 Retry (повтор)** — агент пробует снова через 30 секунд, максимум 3 попытки.

**🟠 Alternative (альтернатива)** — если основной источник недоступен → переключаемся на запасной.

**🔴 Graceful degradation** — если ничего не работает → выполняем упрощённую версию задачи и уведомляем.

### prompt
```
При выполнении задачи [ЗАДАЧА] используй следующие fallback стратегии: Уровень 1 — при ошибке повтори через 30 секунд (максимум 3 раза). Уровень 2 — если 3 попытки не удались → используй альтернативный источник [АЛЬТЕРНАТИВА]. Уровень 3 — если и альтернатива недоступна → выполни упрощённую версию [УПРОЩЁННАЯ ЗАДАЧА] и уведоми меня: «Система работает в режиме fallback, [ОПИСАНИЕ] недоступно.»
```

### steps
1. Определи самые критичные точки своей системы
2. Для каждой точки: что такое Retry, Alternative, Graceful degradation?
3. Добавь fallback промпт в своих агентов
4. Протестируй: дай агенту заведомо неработающий источник данных
5. Убедись что fallback уведомляет тебя с понятным описанием проблемы

### checklist
- Определены критические точки системы
- Для каждой есть минимум 2 уровня fallback
- Тест на сбой прошёл — система восстановилась и уведомила

### troubleshoot
**«Не знаю что использовать как альтернативный источник»** → Запроси у агента: «Назови 3 альтернативных способа получить данные о [X] если основной источник недоступен.»

**«Система зацикливается на retry»** → Добавь счётчик явно: «После 3 попыток — СТОП, переходи к уровню 2. Не пробуй ещё раз.»
