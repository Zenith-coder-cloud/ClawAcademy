# Block 3 — Контент-спек Part 2 (уроки 18–20)

## УРОК 18 — Границы агента: что можно и что нельзя (продолжение)
**track:** all | **timer:** 15 мин | **prev:** 17 | **next:** 19

### infoBlocks
**📁 Доступ к файлам** — только нужные папки, не весь диск.

**🌐 Доступ к сети** — только нужные сайты и API, не весь интернет.

**💬 Контекст** — только информация необходимая для задачи, не весь разговор.

**🔑 Ключи и секреты** — агент не знает твои пароли если они не нужны для его задачи.

### prompt
```
Ты — агент для задачи [ЗАДАЧА]. Твои разрешённые действия: [СПИСОК]. Тебе запрещено: [ЗАПРЕТЫ]. Если тебе понадобится что-то за пределами этих прав — сообщи мне и жди подтверждения. Не действуй самостоятельно за пределами своей зоны.
```

### steps
1. Возьми список своих агентов из предыдущих уроков
2. Для каждого агента напиши: «Разрешено: [...]. Запрещено: [...]»
3. Добавь эти ограничения в промпты агентов
4. Проверь: попроси агента сделать что-то за пределами его зоны — должен отказать
5. Убедись что вся система работает с ограниченными правами

### checklist
- Для каждого агента определены разрешения и запреты
- Ограничения добавлены в промпты
- Агент отказывается действовать за пределами своей зоны

### troubleshoot
**«Агент игнорирует ограничения»** → Сделай запрет более явным: «Это АБСОЛЮТНЫЙ запрет. При любой попытке выйти за эти рамки — останови и напиши: ТРЕБУЕТСЯ ПОДТВЕРЖДЕНИЕ ХОЗЯИНА.»

**«Слишком много ограничений замедляет работу»** → Начни с одного самого важного ограничения для каждого агента. Добавляй постепенно.

## УРОК 19 — Секреты и API ключи: безопасное хранение
**track:** all | **timer:** 15 мин | **prev:** 18 | **next:** 20

### whyNeeded
API ключи, пароли, токены — это твои деньги и твоя безопасность. Случайная утечка ключа может стоить очень дорого. Есть несколько простых правил которые защищают от 95% рисков — и ни одно из них не требует специальных знаний.

### case
**Разработчик Вася** написал ключ API прямо в тексте задачи агенту. Агент записал его в свою историю. Через месяц он поменял провайдера и не удалил старые записи. Ключ оказался в открытом доступе. За ночь — $800 использованных API запросов от чужих ботов. Простое правило «никогда не пиши ключи в текст задачи» — предотвратило бы это.

### capabilities
- Знать правила безопасного хранения секретов
- Передавать ключи агентам безопасным способом
- Настроить ротацию ключей

### infoBlocks
**❌ Никогда:** не пиши ключи прямо в тексте задачи, не храни ключи в историях разговоров, не передавай ключи через незашифрованные каналы.

**✅ Правильно:** храни ключи в переменных окружения (.env файл), передавай агенту через системный промпт (не в диалоге), используй минимально необходимые права для каждого ключа.

**🔄 Ротация:** меняй ключи каждые 90 дней или сразу при малейшем подозрении на утечку.

### prompt
```
При работе с этой задачей тебе потребуется API ключ. Я передам его тебе через системные настройки — не через этот диалог. Если тебе нужен ключ для выполнения задачи — сообщи мне об этом, не пытайся угадать или использовать placeholder.
```

### steps
1. Проверь: есть ли в твоих текущих промптах ключи написанные прямо в тексте?
2. Если да — удали их из текстов, перенеси в системные настройки
3. Для каждого ключа проверь: какой минимальный уровень доступа нужен?
4. Установи напоминание для ротации ключей раз в 90 дней
5. Проверь что агенты не логируют секреты в свои ответы

### checklist
- Нет ни одного ключа в открытом тексте задач или диалогах
- Ключи хранятся в системных настройках или .env
- Настроена ротация ключей

### troubleshoot
**«Не знаю как передать ключ агенту безопасно»** → В OpenClaw: добавь ключ в системный промпт агента через настройки, не через диалог. Агент получит доступ к нему без записи в историю.

**«Уже есть ключи в старых разговорах»** → Отзови старые ключи на стороне провайдера, создай новые. Старые диалоги уже ничего не стоят — важно чтобы ключи были неактивными.

## УРОК 20 — Капстоун: запусти свою первую мультиагентную систему
**track:** all | **timer:** 45 мин | **prev:** 19 | **next:** null

### whyNeeded
Это финальный урок Block 3. Ты прошёл путь от «зачем вообще мультиагент» до полного понимания архитектуры, ролей, делегирования, мониторинга и безопасности. Сейчас — собрать всё вместе и запустить реальную систему которая будет работать для тебя каждый день.

### case
**Ты уже сделал всё по частям** в предыдущих уроках. Агенты настроены, промпты написаны, cron-расписание продумано, watchdog знает что делать при сбое. Этот урок — финальная сборка и первый полноценный запуск. После него у тебя будет работающая система — не тестовая, а настоящая.

### capabilities
- Собрать полную схему своей мультиагентной системы
- Запустить систему и провести первый боевой тест
- Зафиксировать результат и настроить регулярную работу

### infoBlocks
**Твоя система зависит от трека:**

🧑‍💼 **Фрилансер:** КП-агент + Трекер + Коммуникатор + Биллинг → Orchestrator → Watchdog

🏢 **Бизнес:** CRM + Поддержка + Аналитик → Воронка → Orchestrator → Watchdog

📢 **Контент:** Researcher + Writer + Редактор + Publisher + Дистрибьютор → Orchestrator → Watchdog

💰 **Заработок:** Мониторинг + Планировщик возможностей → Orchestrator → Watchdog

**Чеклист готовности системы:**
- [ ] Все агенты имеют промпты с чёткими Input/Output
- [ ] Orchestrator знает кому что делегировать
- [ ] Cron-расписание настроено для автономной работы
- [ ] QA-агент проверяет результаты перед тем как они дойдут до тебя
- [ ] Watchdog следит за системой и присылает heartbeat
- [ ] Fallback стратегии прописаны для критических точек
- [ ] Ключи и секреты хранятся безопасно
- [ ] Границы каждого агента определены

### prompt
```
Ты — Orchestrator моей мультиагентной системы. Вот полная схема: [СХЕМА СИСТЕМЫ]. Начинай работу. Первая задача: [ЗАДАЧА]. Докладывай мне только финальный результат или если нужно моё решение.
```

### steps
1. Открой схему системы которую сохранил в уроке 7 — обнови её
2. Проверь каждый пункт чеклиста готовности выше
3. Сделай первый боевой запуск: дай Orchestrator'у реальную задачу
4. Наблюдай за работой — проверь heartbeat, QA-результаты, финальный вывод
5. Зафиксируй: система работает. Настрой cron для ежедневного/еженедельного запуска

### checklist
- Все агенты системы настроены и протестированы
- Первый боевой запуск прошёл успешно
- Cron настроен — система работает без твоего участия
- Watchdog прислал первый heartbeat

### troubleshoot
**«Система слишком сложная — не знаю с чего начать финальную сборку»** → Начни с малого: Orchestrator + 2 Worker'а. Запусти их. Добавь остальных по одному.

**«Первый запуск не прошёл»** → Это нормально. Посмотри что именно пошло не так: агент не понял задачу? Неправильный формат? Применяй уроки 16–17 — fallback и восстановление.

**«Не знаю как измерить что система работает хорошо»** → Определи один главный метрик. Для фрилансера — часы сэкономленные в неделю. Для бизнеса — % вопросов которые обработала поддержка. Для контента — посты в неделю без твоего ручного труда. Для заработка — количество возможностей которые нашёл агент.
